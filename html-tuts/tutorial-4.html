[0:45 - 1:30]

What are HTML Comments?

HTML comments are pieces of text in your code that are not displayed in the browser. They're only visible when looking at the source code.

Comments start with <!-- and end with -->. Anything between these markers is completely ignored by the browser.

<!-- This is an HTML comment -->
<h1>This heading IS visible</h1>
<!-- The heading above will appear on the webpage -->
<!-- This text is invisible to users -->
<h1>This text IS visible to users</h1>
<!-- Another comment explaining the code -->

Browser Behavior

When a browser reads your HTML, it completely skips anything between <!-- and -->. These comments are only for human readers - you, other developers, or your future self!

[1:30 - 2:45]

Why Use Comments?

Comments serve several important purposes in professional web development:

❌ Without Comments

• Mysterious, hard-to-understand code

• Future you won't remember why you did things

• Team members get confused

• Difficult to maintain and update

✅ With Good Comments

• Clear, understandable code

• Future you knows exactly what's happening

• Team collaboration is smooth

• Easy to maintain and update

5 Main Reasons to Comment

  1. Explain complex code: Help others (and your future self) understand what tricky code does
  2. Section organization: Divide your HTML into logical sections
  3. Temporarily disable code: "Comment out" code you don't want to delete but don't want to run
  4. Leave TODOs: Mark areas that need work or improvements
  5. Documentation: Provide information about the file, author, or last update

Team Development Reality

In professional web development, teams of developers work on the same codebase. Good comments help everyone understand each other's code without having to ask questions constantly.

It's like leaving a map for others who will travel the same path later. The time you spend writing comments today saves hours of confusion tomorrow!

[2:45 - 4:00]

Types of HTML Comments

There are several ways to use comments in HTML. Let's look at the most common types:

<!-- 1. Single-line comment -->
<h1>Welcome</h1>

<!-- 2. Multi-line comment
    This comment spans
    multiple lines -->

<!-- 3. Inline comment -->
<p>This is a paragraph <!-- with an inline comment --> with text.</p>

<!-- 4. Section divider comment -->
<!-- =============== HEADER SECTION =============== -->
<header>...</header>

<!-- 5. TODO comment -->
<!-- TODO: Add contact form here -->
<!-- FIXME: This causes mobile layout issues -->

VS Code Comment Shortcuts

In Visual Studio Code, you can use keyboard shortcuts to comment code quickly:

  • Windows/Linux: Ctrl + / (slash)
  • Mac: Cmd + / (slash)

Select any code and press this shortcut to automatically wrap it in comment tags. Press again to remove the comments!

[DEMONSTRATE] Show this shortcut in action in VS Code.

Commenting Best Practices

  • Be clear and concise: Don't write essays, but be descriptive enough
  • Use proper grammar: Treat comments like documentation
  • Update comments when you update code: Outdated comments are worse than no comments
  • Use TODO/FIXME consistently: These are searchable markers for future work
  • Don't state the obvious: <!-- This is a heading --> before <h1> is unnecessary
[4:00 - 5:00]

Practical Comment Examples

Let's look at some practical examples of good commenting in real HTML code:

<!-- ============================================ -->
<!-- FILE: index.html -->
<!-- AUTHOR: [Your Name] -->
<!-- CREATED: 2023-10-15 -->
<!-- DESCRIPTION: Main landing page for website -->
<!-- ============================================ -->

<!DOCTYPE html>
<html lang="en">

<!-- HEAD SECTION: Metadata only -->
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>My Website</title>

  <!-- External CSS file -->
  <link rel="stylesheet" href="styles.css">
</head>

<!-- BODY SECTION: All visible content -->
<body>

  <!-- MAIN HEADER -->
  <header>
    <h1>Welcome</h1>
  </header>

  <!-- TODO: Add navigation menu here -->
<!-- <nav>...</nav> -->

  <!-- MAIN CONTENT AREA -->
  <main>
    <article>
      <h2>About Us</h2>

      <!-- FIXME: This paragraph needs updating -->
      <p>Company information goes here.</p>
    </article>
  </main>

  <!-- FOOTER -->
  <footer>
    <p>© 2023 My Site</p>
  </footer>

  <!-- JavaScript for interactivity -->
  <script src="script.js"></script>
</body>
</html>

Professional Benefits

This commented code shows several professional practices:

  1. File header with metadata (author, date, purpose)
  2. Section dividers to organize large files
  3. TODO markers for planned features
  4. FIXME markers for known issues
  5. Explanatory comments for non-obvious code

This makes the code self-documenting - anyone can understand it without asking the original author!

[5:00 - 5:30]

Commenting Out Code

One of the most useful features of comments is the ability to temporarily disable code without deleting it. This is called "commenting out" code.

<!-- Old navigation - keeping for reference -->
<!--
<nav>
  <a href="#">Home</a>
  <a href="#">About</a>
</nav>
-->

<!-- New navigation -->
<nav>
  <a href="#">Home</a>
  <a href="#">About</a>
  <a href="#">Contact</a> <!-- New! -->
</nav>

When to Comment Out Code

  • Debugging: Temporarily remove code to isolate problems
  • A/B testing: Try different versions of a feature
  • Keeping old code for reference: Don't delete, just comment out
  • Temporary features: Holiday themes, special promotions
  • Incomplete work: Save work in progress without breaking the site

Remember: Don't leave large blocks of commented-out code in production. Use version control (like Git) for keeping history instead.

[5:30 - 5:45]

Common Comment Mistakes

Commenting Mistakes to Avoid

Mistake 1: Over-commenting (stating the obvious)
Bad: <!-- This is a paragraph --> before <p>
Good: Only comment when it adds real value

Mistake 2: Under-commenting complex code
Bad: No comments on tricky algorithm or workaround
Good: Explain why you did something unusual

Mistake 3: Outdated comments
Bad: Comments that no longer match the code
Good: Update comments when you update code

Mistake 4: Rude or unprofessional comments
Bad: <!-- This is stupid but it works -->
Good: Keep comments professional and helpful

Mistake 5: Nested comments (not allowed in HTML)
Bad: <!-- Comment <!-- inside --> comment -->
Good: HTML doesn't allow nested comments. Use single comments.

The Goldilocks Principle

Good commenting is like Goldilocks' porridge: not too much, not too little, but just right.

• Too few comments: Code is hard to understand
• Too many comments: Code is cluttered and hard to read
• Just right: Comments explain the "why" not the "what"
