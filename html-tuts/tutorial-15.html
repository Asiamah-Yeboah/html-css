Basic Form Structure

Every HTML form starts with the <form> element. Here's the basic structure:

<!-- Basic form structure -->
<form action="/submit-form" method="POST">

  <!-- Form controls go here -->
  <label for="name">Name:</label>
  <input type="text" id="name" name="name">

  <button type="submit">Submit</button>
</form>

Key Form Attributes:

  • action: Where the form data is sent (URL)
  • method: HTTP method to use (GET or POST)
  • GET: Data visible in URL (for searches)
  • POST: Data hidden (for passwords, sensitive info)
  • name: Identifies data on server (required!)

Input Types Gallery

HTML5 introduced many new input types that improve user experience and validation. Here are the most important ones:

Text Input

<input type="text">

Single line text input. Most common form element.

Email Input

<input type="email">

Validates email format automatically.

Password Input

<input type="password">

Hides characters with dots or asterisks.

Telephone Input

<input type="tel">

Optimized for phone numbers on mobile.

URL Input

<input type="url">

Validates website URLs.

Date Input

<input type="date">

Shows date picker on supported browsers.

Search Input

<input type="search">

Styled for search with clear button.

Range Input

<input type="range">

Slider for selecting a value.

Number Input

<input type="number">

Only accepts numbers with increment buttons.

Mobile Optimization

Different input types trigger different mobile keyboards! type="email" shows @ key, type="tel" shows number pad, type="url" shows .com key. This small detail greatly improves mobile user experience!

Form Controls Deep Dive

Beyond basic inputs, forms have specialized controls for different data types:

<!-- Complete form with various controls -->
<form>

  <!-- Textarea for multi-line text -->
  <label for="message">Message:</label>
  <textarea id="message" name="message" rows="4" cols="50">
    Default text here...
  </textarea>

  <!-- Select dropdown -->
  <label for="country">Country:</label>
  <select id="country" name="country">
    <option value="">Select a country</option>
    <option value="us">United States</option>
    <option value="uk">United Kingdom</option>
    <option value="ca">Canada</option>
  </select>

  <!-- Radio buttons (single selection) -->
  <fieldset>
    <legend>Preferred Contact Method:</legend>
    <input type="radio" id="contact-email" name="contact" value="email">
    <label for="contact-email">Email</label>
    <br>
    <input type="radio" id="contact-phone" name="contact" value="phone">
    <label for="contact-phone">Phone</label>
  </fieldset>

  <!-- Checkboxes (multiple selection) -->
  <input type="checkbox" id="newsletter" name="newsletter">
  <label for="newsletter">Subscribe to newsletter</label>

  <button type="submit">Submit</button>
  <button type="reset">Reset</button>
</form>

HTML5 Form Validation

HTML5 introduced built-in validation attributes that work without JavaScript!

Try submitting without filling this
Try entering an invalid email
Must match pattern: 123-456-7890
<!-- Common validation attributes -->
<input type="text" required>
<!-- Field must be filled -->

<input type="email" required>
<!-- Must be valid email format -->

<input type="text" minlength="6">
<!-- At least 6 characters -->

<input type="text" maxlength="10">
<!-- Maximum 10 characters -->

<input type="text" pattern="\d{3}-\d{3}-\d{4}">
<!-- Must match regex pattern -->

Validation Best Practices

  • Always validate on server too - client-side validation can be bypassed
  • Show helpful error messages - use title attribute or custom validation
  • Validate as user types - use JavaScript for real-time feedback
  • Don't over-validate - only require what's truly necessary
  • Test without JavaScript - ensure basic HTML validation works

Form Accessibility

Forms can be challenging for users with disabilities. Here's how to make them accessible:

‚ùå Inaccessible Form

Name:
<input type="text">
<br>
Subscribe:
<input type="checkbox">

Problems: No labels, screen readers can't associate labels with inputs, keyboard navigation fails

‚úÖ Accessible Form

<label for="name">Name:</label>
<input type="text" id="name">
<br>
<input type="checkbox" id="subscribe">
<label for="subscribe">Subscribe</label>

Benefits: Screen readers announce labels, keyboard navigation works, clear associations

Accessibility Checklist

  • Always use <label> elements with for attributes
  • Group related fields with <fieldset> and <legend>
  • Use proper input types for mobile optimization
  • Add ARIA attributes for complex forms (aria-describedby, aria-invalid)
  • Ensure keyboard navigation - Tab order should be logical
  • Provide clear instructions and error messages
  • Test with screen readers like NVDA or VoiceOver
[9:30 - 10:00]

Common Mistakes to Avoid

Form Mistakes Beginners Make

  • Missing labels: The #1 accessibility violation
  • Too many required fields: Reduces completion rates
  • Poor error messages: "Invalid input" instead of helpful guidance
  • No validation feedback: Users don't know what's wrong
  • Ignoring mobile users: Tiny buttons, wrong keyboards
  • Security issues: Client-side only validation
  • Bad tab order: Confusing keyboard navigation
  • Forgetting the name attribute: Data won't be sent to server

Form Optimization Tip

Studies show that reducing a form from 4 fields to 3 can increase conversions by 50%. Every field you remove increases completion rate. Only ask for what you absolutely need!
